## リサーチ根拠

### 間隔反復（Spaced Repetition）の効果
- Cepeda et al. (2006): 間隔を空けた復習は集中学習に比べて **長期記憶保持率が最大200%向上**
- Leitner システム: 正解した問題の復習間隔を延ばし、不正解の問題は短い間隔で再出題する
- 簿記トレでは AdaptiveQuestionSelector が「弱点 > 未挑戦 > 復習 > 得意」の優先度で出題

### テスト効果（Testing Effect / Retrieval Practice）
- Roediger & Butler (2011): テスト（問題を解く行為）自体が記憶を強化する — 単なる読み返しより **50-70%** 効果的
- 誤答後のフィードバックが特に重要: 間違えた直後に正解と解説を見ることで修正学習が起こる
- 簿記トレの差分学習（不正解時に UserAnswer vs CorrectAnswer を並べて比較表示）はこの原則に合致

### 自己決定理論（SDT: Deci & Ryan, 2000）— 学習動機の3要素
| 要素 | 定義 | 簿記トレでの実現手段 |
|------|------|----------------------|
| 自律性 | 自分で選んでいる感覚 | カテゴリ選択、学習プラン設定、出題数の自由選択 |
| 有能感 | 「できるようになった」実感 | コンボ表示、正解率推移、実力診断ランク |
| 関係性 | 他者とのつながり | ストリーク共有（※現在未実装、将来拡張候補） |

### 認知負荷理論（Sweller, 1988）
- **内在的負荷**: 簿記の概念そのものの難しさ → 用語解説（TermDefinition）で軽減
- **外在的負荷**: UIの複雑さ → 1画面1問、フォーマット別に最適化されたレイアウト
- **学習関連負荷**: 問題を解く過程の思考 → 適切な難易度出題（difficulty 1-3）で維持

### 試験合格率データ（日本商工会議所）
- 日商簿記3級合格率: 約 **40-50%**（回によって変動）
- CBT方式導入後（2020年〜）: 受験機会の増加により学習計画の柔軟性が向上
- 合格ボーダー: 70点/100点 → 「完璧主義」ではなく「合格点到達」を目標設計に反映

## オンボーディング固有基準

### パーソナライゼーションの焦点
- 「簿記の学習経験」→ 初学者/学習中/再挑戦 で初期難易度を調整
- 「試験日の目標」→ 学習プラン（1日あたりの目標問題数）に反映
- 「1日の学習時間」→ 通知頻度・出題数の推奨値に反映
- 診断テスト結果 → 弱点カテゴリの優先出題に反映

### 初回体験の黄金法則
- **最初の3問で「解けた！」体験を作る**: 難易度1の仕訳問題から開始
- 初回セッションは必ず正解フィードバック（コンボ演出、ハプティクス）を体験させる
- 解説の有用性を初回で示す: 「なるほど、こういう仕組みか」の理解体験

### 注意: ペイウォールの配置
- オンボーディング中にペイウォールを複数回表示しない（現状CR-1として検出済み）
- コアアクション（問題を解く → 解説を読む → 成長を実感）を体験した**後**にペイウォールを1回だけ表示
- 無料版でも十分に学習できることを先に示し、Premium の付加価値（広告非表示 + AI分析）で訴求

## コアループ固有基準

### コアループの理想形
通知 → アプリ起動 → 問題表示 → 解答 → 正誤フィードバック → 解説確認 → 次の問題 or 結果画面 → 閉じる

### スキマ時間学習の最適化
- 1問あたりの解答時間目安: 30-90秒（フォーマットにより変動）
- 5問セット = 約5分、10問セット = 約10分 で完結
- セッション中断時の自動保存（QuizSessionPersistence）で途中再開可能
- デイリーチャレンジ: 毎日3問の少量タスクで「とりあえず今日もやった」感を生む

### 変動報酬の設計
- **出題の変動性**: AdaptiveQuestionSelector による弱点重視の適応型出題（毎回異なる組み合わせ）
- **コンボシステム**: 連続正解でコンボ数表示、3の倍数で追加フィードバック（過剰演出を避ける）
- **デイリーチャレンジ**: 日替わり3問で「今日は何が出るか」の期待感
- **実力診断**: 定期的にランク判定（合格圏/あと一歩/基礎固め等）で現在地を可視化

### フォーマット別UX要件
| フォーマット | 試験セクション | 操作 | UX注意点 |
|-------------|---------------|------|----------|
| multipleChoice | - | 4択タップ | 選択肢が正解を必ず含むことを保証 |
| cbtJournalEntry | 第1問 | 勘定科目選択 + 金額入力 | numberPad に「完了」ボタン必須 |
| tAccountFill | 第2問(1) | T字勘定の空欄補充 | 勘定科目Picker + 金額入力、空欄ラベルの統一（blank_1→空欄(1)） |
| subledgerSelect | 第2問(2)-A | 補助簿チェックボックス | 複数選択可、全チェック後に解答ボタン有効化 |
| theoryFill | 第2問(2)-B | プルダウン語句選択 | 解答後にUIが消えて空白にならないこと |

### Peak-End 設計
- **Peak**: 連続正解時のコンボ演出 + ハプティクス / デイリーチャレンジ達成時の CelebrationOverlay
- **End**: 結果画面でのランク表示 + 正解率 + 次回への励ましメッセージ + 弱点カテゴリの提示

## チャーンリスク固有要因

| リスク | 原因 | 対策 |
|--------|------|------|
| Day 0 離脱 | オンボーディング中の2重ペイウォール | ペイウォールを1箇所に絞り、コアアクション体験後に配置 |
| Day 1-3 離脱 | 「簿記が難しい」「問題の意味が分からない」 | 難易度1から開始 + 用語解説（TermDefinition）+ 身近な例え |
| Day 3-7 離脱 | 通知が画一的で開く動機にならない | パーソナライズされた通知文（進捗ベース + 感情的訴求のバリエーション） |
| Day 7-14 離脱 | 「正解率が上がらない」→ 自己効力感の低下 | 弱点カテゴリの可視化 + 「前回より○%向上」の成長フィードバック |
| Day 14-30 離脱 | マンネリ化（同じフォーマットの繰り返し） | 新フォーマットの段階的開放 + 実力診断でランクアップ体験 |
| Day 30+ 離脱 | 「合格点に届かない」→ 試験への不安 | 模擬試験モード + セクション別得点率 + 合格ラインとの差分可視化 |
| 試験後離脱 | 合格して目的達成 / 不合格で挫折 | 合格者: 2級への導線 / 不合格者: 弱点分析 + リベンジプラン提示 |
| 広告ストレス離脱 | インタースティシャル広告の頻度が高すぎる | 学習フロー中は表示しない、結果画面後のみ、5問ごと等の間隔制御 |

## コンテンツ品質基準

### 問題文の日本語品質
- 実際の試験問題に近い文体を使用（「次の取引を仕訳しなさい」）
- 勘定科目名は日商簿記3級の公式勘定科目表に準拠
- 金額は現実的な範囲（¥100〜¥10,000,000）で、端数が出にくい値を使用
- NGパターン: 実在の企業名の使用、非現実的な取引内容、曖昧な指示文

### 解説の品質基準
| 基準 | 内容 | 検証方法 |
|------|------|----------|
| 段階的説明 | 「なぜこの仕訳になるか」を論理的に分解 | 初学者が読んで「なるほど」と思えるか |
| 差分強調 | 不正解時にユーザー回答 vs 正解を視覚比較 | UserAnswer と CorrectAnswer の並列表示 |
| 用語解説 | 専門用語に初心者向け定義 + 身近な例え | TermDefinition の realWorldExample が自然か |
| 出題頻度 | frequencyRank (A/B/C) で重要度を明示 | 「本試験で頻出」の情報が学習の優先度判断を助けるか |
| 試験セクション | examSection で本試験のどこに対応するか明示 | 「第1問対策」等のセクション学習が可能か |

### 用語解説（TermDefinition）の品質
- **term**: 正式な簿記用語（「売掛金」「減価償却累計額」等）
- **category**: 5要素分類（資産/負債/純資産/収益/費用）に準拠
- **definition**: 1-2文で簡潔に。「〜のこと」で終わる辞書的説明ではなく、概念の本質を伝える
- **realWorldExample**: 日常生活や身近なビジネスシーンに置き換えた具体例（「ツケで買い物したときの未払い分」等）

## 感情デザイン固有基準

### 学習アプリ特有の感情曲線
```
起動 → 少し緊張（「今日は解けるかな」）
  → 問題表示 → 集中（フロー状態）
    → 正解 → 達成感（コンボ・ハプティクス）
    → 不正解 → 軽い悔しさ → 解説で納得（「なるほど！」）
      → 結果画面 → 自己成長の実感（「前回より上がった」）
        → 閉じる → 「明日もやろう」
```

### 不正解時の感情ケア（最重要）
- 間違えることは学習の一部 → **罰ではなく学びの機会**として演出
- 不正解時の赤色は最小限（全画面赤にしない）、解説セクションは中立色で表示
- 「惜しい！」「次はいける！」等のポジティブメッセージ（MotivationMessageView）
- 不正解問題は自動でブックマーク候補として提示（復習への橋渡し）

### 有能感（コンピテンス）の設計
- **マイクロ達成**: 1問正解ごとのフィードバック（チェックマーク + 効果音）
- **セッション達成**: 結果画面でのランク表示（S/A/B/C/D）+ 正解率
- **累積達成**: ストリーク（連続学習日数）+ 実力診断ランク + バッジシステム
- **成長の可視化**: カテゴリ別マスタリー率 + 弱点→得意への推移グラフ

### 色彩設計
| 用途 | 色 | 意図 |
|------|------|------|
| 正解 | appPrimary（青緑系） | 達成感・信頼感。赤の反対で視覚的に明快 |
| 不正解 | appError（赤系） | 注意喚起。ただし面積を最小限に抑えて威圧感を減らす |
| 進捗 | appOrange（オレンジ系） | 温かみ・前進感。ゲーミフィケーション要素に使用 |
| 背景 | 白〜薄グレー | 長時間学習でも目が疲れにくい。教科書的な清潔感 |

### 学習者の心理特性（日本の資格試験文化）
- 「落ちたらどうしよう」という不安が強い → 合格ラインとの距離を**ポジティブに**可視化（「あと10点！」ではなく「70%到達済み！」）
- 完璧主義傾向 → 「70点で合格」を繰り返し伝え、全問正解を目指す必要がないことを示す
- 他人と比較されたくない → ランキング機能は慎重に（自己ベストとの比較を主軸に）
- 「勉強した感」が欲しい → 学習時間・問題数の統計表示で投資の可視化
